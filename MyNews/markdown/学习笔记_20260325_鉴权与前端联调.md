# 学习笔记（2026-03-25）

> 主题：JWT 鉴权落地 + 前端联调 + 组件化与复用

## 1. 今天到底做了什么（高层概览）

1. 后端把登录改造成 JWT 登录，返回 `access_token`。
2. 后端给敏感接口加了依赖鉴权（`Depends(get_current_user)`）。
3. 前端把登录弹窗、顶部导航抽成公共组件。
4. 前端把跨页面重复逻辑下沉到 `useTopNavAuth`。
5. 前端入口增加 axios 拦截器，自动带 `Authorization: Bearer <token>`。
6. 修复了一个常见交互问题：输入框回车提交不稳定（从按钮按键事件改为 form submit）。

## 2. 重点一：后端 JWT 验证链路（结合代码）

### 2.1 登录签发 Token

文件：`MyNews/App/routers/users.py`

关键逻辑（简化）：

```python
@router.post("/login", response_model=ApiResponse[LoginTokenData])
async def login(user_in: UserLogin, db: AsyncSession = Depends(get_db)):
    db_user = await user.get_user_by_username(db, username=user_in.username)
    if not db_user or not verify_password(user_in.password, db_user.password):
        raise HTTPException(status_code=400, detail="用户名或密码错误")

    access_token, expires_in = create_access_token(db_user.id, db_user.username)
    token_data = LoginTokenData(
        access_token=access_token,
        token_type="bearer",
        expires_in=expires_in,
        user=UserOut.from_orm(db_user),
    )
    return success_response(token_data, message="登录成功")
```

你要理解的点：

1. 登录成功不仅返回用户信息，还返回 token。
2. 返回结构固定：`access_token + token_type + expires_in + user`。
3. 前端后续访问受保护接口必须传这个 token。

### 2.2 Token 内容与签名

文件：`MyNews/App/utils/jwt_auth.py`

关键逻辑（简化）：

```python
SECRET_KEY = os.getenv("JWT_SECRET_KEY", "dev-only-secret-change-me-at-least-32-bytes")
ALGORITHM = "HS256"

def create_access_token(user_id: int, username: str) -> tuple[str, int]:
    payload = {
        "sub": str(user_id),
        "username": username,
        "exp": expire_at,
    }
    token = jwt.encode(payload, SECRET_KEY, algorithm=ALGORITHM)
    return token, ACCESS_TOKEN_EXPIRE_MINUTES * 60
```

你要理解的点：

1. `sub` 放的是用户 ID（后续鉴权核心字段）。
2. `exp` 是过期时间，过期后会 401。
3. `SECRET_KEY` + `ALGORITHM` 决定签名是否可信。

### 2.3 受保护接口如何验 Token

文件：`MyNews/App/dependencies/auth.py`

关键逻辑（简化）：

```python
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/users/login")

async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
    payload = decode_access_token(token)
    user_id = int(payload.get("sub"))
    db_user = await user_crud.get_user_by_id(db, user_id)
    return db_user
```

你要理解的点：

1. `oauth2_scheme` 会从请求头提取 Bearer token。
2. `decode_access_token` 负责验签和过期校验。
3. token 合法还不够，数据库里用户存在才算通过。

### 2.4 401 与 403 的真实区别

文件：`MyNews/App/routers/users.py`

```python
current_user: User = Depends(get_current_user)
if current_user.id != user_id:
    raise HTTPException(status_code=403, detail="无权限操作该用户")
```

结论：

1. 401：身份无效（没 token、token 错、token 过期）。
2. 403：身份有效，但没权限做这个操作。

## 3. 重点二：前端登录态与 Token 使用（结合代码）

### 3.1 登录弹窗如何拿 token

文件：`MyNews/frontend/src/components/AuthModal.vue`

```vue
const res = await axios.post(url, authForm.value)
if (isLoginMode.value && res.data && res.data.code === 200) {
  const loginData = res.data.data || {}
  emit('success', {
    user: loginData.user || null,
    accessToken: loginData.access_token || ''
  })
}
```

你要理解的点：

1. 弹窗不直接写 localStorage，而是发 `success` 事件。
2. 页面或 composable 统一接收这个事件处理状态。

### 3.2 统一保存登录态

文件：`MyNews/frontend/src/composables/useTopNavAuth.js`

```js
const handleAuthSuccess = ({ user, accessToken }) => {
  currentUser.value = user || null
  if (accessToken) {
    localStorage.setItem('accessToken', accessToken)
  }
  localStorage.setItem('currentUser', JSON.stringify(currentUser.value))
}
```

你要理解的点：

1. `accessToken` 存本地用于后续请求头注入。
2. `currentUser` 存本地用于刷新后恢复 UI 登录态。

### 3.3 为什么以前“登录了也不能访问受保护接口”

核心原因：只保存了 token，没把 token 放进请求头。

现在的修复：

文件：`MyNews/frontend/src/main.js`

```js
axios.interceptors.request.use((config) => {
  const token = localStorage.getItem('accessToken')
  if (token) {
    config.headers = config.headers || {}
    if (!config.headers.Authorization) {
      config.headers.Authorization = `Bearer ${token}`
    }
  }
  return config
})
```

你要理解的点：

1. 这是“全局生效”的关键，一次配置，所有 axios 请求都会带 token。
2. 如果请求手动设置了 `Authorization`，拦截器不会覆盖。

## 4. 重点三：组件化与复用（不是只抽 UI）

### 4.1 TopNavBar 为何只发事件，不直接写 logout/goHome

文件：`MyNews/frontend/src/components/TopNavBar.vue`

```vue
const emit = defineEmits(['logo-click', 'login-click', 'profile-click', 'logout-click', 'publish-click'])
```

设计原因：

1. 组件层只关心展示与交互信号，保持可复用。
2. 业务动作（跳路由、清 localStorage）放在 composable 更好复用。

### 4.2 业务复用层：useTopNavAuth

文件：`MyNews/frontend/src/composables/useTopNavAuth.js`

这里统一收敛了跨页面重复逻辑：

1. 恢复登录态：`restoreCurrentUser`
2. 登录成功写入：`handleAuthSuccess`
3. 退出登录：`logout`
4. 发布按钮行为：`handlePublishClick`（未登录先弹登录）
5. 页面鉴权守卫：`ensureAuthenticated`

收益：

1. Home、NewsDetail、PublishNews 三页不再重复写一堆函数。
2. 改登录态策略时只改一个文件。

## 5. 关键流程图（文字版）

### 5.1 登录流程

1. 点击导航栏“登录” -> TopNavBar 发 `login-click`。
2. 页面把 `showAuthModal` 设为 true。
3. AuthModal 提交 `/users/login`。
4. 后端验证账号密码，返回 token。
5. 前端 `handleAuthSuccess` 保存 `accessToken + currentUser`。
6. 弹窗关闭，导航栏显示用户名。

### 5.2 调用受保护接口流程

1. 页面发起 axios 请求。
2. `main.js` 请求拦截器自动补 `Authorization`。
3. 后端 `oauth2_scheme` 提取 token。
4. `get_current_user` 验签、验过期、查用户。
5. 路由层再做资源权限判断（如“只能改自己”）。
6. 通过则返回成功；失败则 401/403。

## 6. 今天的典型坑 + 修复方式

### 6.1 坑：登录返回结构变化导致前端解析错位

现象：

1. 后端已登录成功，但前端展示异常。

修复：

1. 前端改为读取 `res.data.data.user` 与 `res.data.data.access_token`。

### 6.2 坑：按钮上 `@keyup.enter` 不稳定

现象：

1. 输入框按回车，登录/注册有时不触发。

根因：

1. 键盘焦点在 input，不在 button。

修复（已完成）：

1. 用 `<form @submit.prevent="submitAuth">`。
2. 按钮改为 `type="submit"`。

### 6.3 坑：只存 token，不发 token

现象：

1. 看起来已登录，但请求受保护接口依然 401。

修复：

1. 在前端入口配置 axios 请求拦截器，自动注入 Bearer token。

## 7. 复盘结论（面试/复习可直接背）

1. JWT 鉴权不是“登录成功”就结束，而是“每次请求都要带 token 并在后端验证”。
2. 组件复用分两层：UI 组件复用 + 业务逻辑复用（composable/store）。
3. 401 与 403 一定要分清，排错速度会快很多。
4. 前端联调最容易漏的是“响应结构改了但解析代码没改”。
5. 输入提交场景优先 form submit，不要只依赖按钮按键事件。

---

## 8. 下一步建议（按优先级）

1. 增加 401 统一处理：token 过期后自动清理并提示重新登录。
2. 封装统一 API 客户端（baseURL、超时、错误映射、拦截器集中管理）。
3. 增加 refresh token 方案，提升长期登录体验。
4. 给鉴权关键路径补测试（登录、过期、无权限、退出后请求）。
