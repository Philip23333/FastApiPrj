# 慢响应分层定位说明（MyNews）

## 1. 目的

本文档说明本次如何对“线上访问明显变慢”进行分层定位，并最终锁定根因。方法可复用到后续性能故障排查。

适用场景：
- 前端页面打开慢、接口返回慢
- 不确定问题在 Nginx、服务器资源、数据库、还是应用代码

---

## 2. 分层定位思路

采用“由外到内、逐层排除”的方式：

1. **入口层（Nginx）**：是否代理异常、超时、5xx
2. **主机层（CPU/内存/负载）**：是否资源打满
3. **网络与端口层**：服务监听是否正确、链路是否绕路
4. **数据库层（连通性/SQL耗时）**：是否数据库慢
5. **应用层（FastAPI/缓存/业务逻辑）**：是否某段代码阻塞

每层都遵循同一原则：
- 先测“现象”
- 再找“证据”
- 最后做“结论”

---

## 3. 第 1 层：入口层（Nginx）排查

### 3.1 检查配置是否合理

执行：

```bash
sudo nginx -T
```

重点确认：
- `location /api/` 正确反向代理到 `127.0.0.1:8080`
- 前端静态资源由 Nginx 托管
- 没有明显错误配置

### 3.2 看 Nginx 日志是否有超时/上游错误

执行：

```bash
sudo tail -n 120 /var/log/nginx/error.log
sudo tail -n 80 /var/log/nginx/access.log
```

判定标准：
- 如果大量出现 `upstream timed out`、`502/504`，问题很可能在 Nginx 到后端链路
- 本次未看到此类上游超时错误，Nginx 并非首要瓶颈

---

## 4. 第 2 层：主机资源排查

### 4.1 看负载与内存

执行：

```bash
uptime
free -m
nproc
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu | head -n 20
```

本次观察：
- Load Average 很低（远未接近 CPU 核心数）
- 内存有可用余量
- 无异常高 CPU 进程

结论：
- 机器资源不是慢响应根因

---

## 5. 第 3 层：网络与端口链路排查

### 5.1 监听状态检查

执行：

```bash
sudo ss -lntp | grep -E ':80|:8080|:3000|:6379'
```

本次确认：
- Nginx 监听 80
- Uvicorn 监听 `127.0.0.1:8080`
- Redis 在 6379 可达

### 5.2 对比“经 Nginx”和“直连后端”耗时

执行：

```bash
# 经 Nginx
curl -s -o /dev/null -w 'code=%{http_code} total=%{time_total}s\n' http://127.0.0.1/api/news/?page=1\&size=10

# 直连后端
curl -s -o /dev/null -w 'code=%{http_code} total=%{time_total}s\n' http://127.0.0.1:8080/news/?page=1\&size=10
```

判定逻辑：
- 若两者都慢，问题在后端应用/依赖，不在 Nginx
- 本次两者都慢（秒级），继续下钻应用层

---

## 6. 第 4 层：数据库排查（连通性 + SQL）

### 6.1 测数据库网络连通时延

执行（TCP 连接延迟）：

```bash
python - <<'PY'
import socket,time
host='rm-wz9eba3q9485y96tzno.mysql.rds.aliyuncs.com'
port=3306
for i in range(5):
    t=time.time(); s=socket.socket(); s.settimeout(3)
    try:
        s.connect((host,port)); ok=True
    except Exception as e:
        ok=False; err=str(e)
    finally:
        s.close()
    dt=(time.time()-t)*1000
    print(i+1, 'ok' if ok else 'fail', f'{dt:.1f}ms', '' if ok else err)
PY
```

本次结果：约 4~5ms，连通性正常。

### 6.2 检查 SQL 执行计划和实际耗时

执行：

```bash
MYSQL_PWD='***' mysql -h <RDS_HOST> -u root -D news_app -e "
EXPLAIN SELECT count(id) FROM news WHERE audit_status='approved' AND is_deleted=0;
EXPLAIN SELECT id,title,views,publish_time,category_id FROM news WHERE audit_status='approved' AND is_deleted=0 ORDER BY publish_time DESC LIMIT 10;
EXPLAIN SELECT id,title,views,publish_time,category_id FROM news WHERE views>=10000 AND audit_status='approved' AND is_deleted=0 ORDER BY views DESC LIMIT 8;
"
```

以及：

```bash
MYSQL_PWD='***' mysql -h <RDS_HOST> -u root -D news_app -e "
SET profiling=1;
SELECT count(id) FROM news WHERE audit_status='approved' AND is_deleted=0;
SELECT id,title,views,publish_time,category_id FROM news WHERE audit_status='approved' AND is_deleted=0 ORDER BY publish_time DESC LIMIT 10;
SELECT id,title,views,publish_time,category_id FROM news WHERE views>=10000 AND audit_status='approved' AND is_deleted=0 ORDER BY views DESC LIMIT 8;
SHOW PROFILES;
"
```

本次结果：SQL 耗时在毫秒级，远小于接口总耗时（秒级）。

结论：
- MySQL 不是本次主要瓶颈。

## 7. 第 5 层：应用层（FastAPI/Redis 缓存）定位

### 7.1 查应用日志关键字

执行：

```bash
sudo tail -n 200 /var/log/fastapi.log
sudo tail -n 120 /var/log/fastapi.err.log
```

发现大量报错：
- `cache read failed ... Authentication required`
- `hot cache read failed ... Authentication required`

说明：
- 缓存访问 Redis 认证失败
- 每次缓存读写失败不是“立即失败”，而是有明显等待，导致接口整体被拖慢

### 7.2 直接测 Redis 调用耗时

执行（应用环境 Python）：

```bash
cd /home/admin/FastApiPrj/MyNews/App && /home/admin/miniconda3/envs/fastapi_tt/bin/python - <<'PY'
import asyncio,time
from config.cache_config import redis_client

async def main():
    for i in range(1,6):
        t=time.time()
        try:
            await redis_client.get('bench:key')
            ok='ok'
        except Exception as e:
            ok=f'err:{type(e).__name__}:{e}'
        print(i, ok, f'{time.time()-t:.3f}s')

asyncio.run(main())
PY
```

本次现象：单次 Redis 失败调用可达数秒到十余秒，和慢接口量级一致。

### 7.3 验证并修复

处理动作：
- 重启后端进程，让当前 `.env` 配置生效

执行：

```bash
sudo supervisorctl restart fastapi-backend
sudo supervisorctl status
```

修复后复测：

```bash
for u in \
  http://127.0.0.1:8080/news/?page=1\&size=10 \
  http://127.0.0.1:8080/news/hot?min_views=10000\&page=1\&size=8 \
  http://127.0.0.1:8080/news/categories; do
  echo "== $u"
  curl -s -o /dev/null -w 'code=%{http_code} total=%{time_total}s\n' "$u"
done
```

本次结果：从秒级降到毫秒级。

额外验证：Redis 已写入缓存 key（`news:list:*`, `news:hot:*`, `news:categories_list:*`）。

---

## 8. 本次定位结论

1. **Nginx 不是根因**：无 upstream 超时特征，代理链路正常。
2. **服务器资源不是根因**：CPU/内存/负载均正常。
3. **MySQL 不是根因**：网络连通与 SQL 执行均快。
4. **根因在应用缓存层**：Redis 鉴权异常导致缓存调用阻塞，拖慢接口。
5. **修复措施有效**：重启后端进程使配置生效后，接口恢复毫秒级响应。

---

## 9. 可复用的排查模板（建议收藏）

1. 先测接口耗时（curl）
2. 看 Nginx access/error
3. 看主机资源（uptime/free/ps）
4. 对比经 Nginx和直连后端
5. 查数据库连通 + SQL profile
6. 查应用日志关键字（timeout/auth/connection）
7. 对可疑依赖做单点基准（如 Redis get）
8. 修复后做前后对比并留证据
